# Branded event QR codes in mvp-app

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `PLANS.md` at the repository root. It covers the mobile side of a cross-repository feature whose backend and web source of truth live in `mvp-site`.

## Purpose / Big Picture

Event hosts need a fast way to show a scannable event link from the mobile event details screen. After this change, the three-dot menu on an event details page includes a QR Code action. Tapping it opens a branded QR image generated by `mvp-site`; scanning that image opens `https://bracket-iq.com/events/{eventId}`, which already routes into the installed app through the existing app links and deep-link handling.

## Decision Log

- Decision: Generate the QR image in `mvp-site` and display that PNG in `mvp-app` instead of generating QR pixels locally in Kotlin.
  Rationale: The backend can centralize the QR payload, logo compositing, access checks, cache headers, and future branding changes. The app already has network image loading dependencies, so it only needs to render the canonical image.
  Date/Author: 2026-05-04 / Codex.

- Decision: Show the mobile QR menu item only for non-template, non-draft, non-private events.
  Rationale: The server QR image route mirrors restricted-event read access, while the app's Coil image request does not carry the app bearer token. Limiting the action to publicly visible events avoids a broken preview for restricted states.
  Date/Author: 2026-05-04 / Codex.

## Context and Orientation

`composeApp/src/commonMain/kotlin/com/razumly/mvp/eventDetail/EventDetailScreen.kt` contains the shared Compose event details UI. Its top-right three-dot menu already has actions such as Edit, Create Template, Join Event, Share, Report Event, Notify Players, and Delete. The existing `Share` item calls `EventDetailComponent.shareEvent()`, which uses `createEventUrl()` from `composeApp/src/commonMain/kotlin/com/razumly/mvp/core/presentation/util/util.kt` to build `https://bracket-iq.com/events/{eventId}`.

The backend route to display is planned as `GET /api/events/{eventId}/qr` in `mvp-site`. The app has `apiBaseUrl` and `getImageUrl()` helpers in the same util file. This plan will add a small event QR URL helper there so the UI does not manually duplicate API path construction.

## Plan of Work

First, add a helper named `getEventQrCodeUrl(eventId: String)` near `getImageUrl()`. It will trim the configured API base URL, append `/api/events/`, append the event id, and append `/qr`.

Next, update `EventDetailScreen.kt` by adding a Boolean state such as `showQrCodeDialog`, a menu item labeled `QR Code`, and a dialog that displays the PNG returned by the backend. The dialog should show the event name, the QR image, the public event URL, and close/share actions. It should use existing Compose and Coil image patterns rather than introducing a local QR library.

Finally, validate that the shared code compiles enough for the change. If the full Android build is too slow or blocked by environment setup, run the most targeted Gradle task available and record the result here.

## Validation and Acceptance

Acceptance is user-visible: open an event detail screen in the app, tap the top-right three-dot menu, tap `QR Code`, and observe a modal with a scannable BracketIQ QR image. The image source must be the mvp-site endpoint `/api/events/{eventId}/qr`. Closing the modal should return to the event details page without changing event state.

## Idempotence and Recovery

The UI changes are additive. If the backend endpoint is unavailable in local development, the image may show a loading or failed state, but the app should remain stable and the dialog should still be dismissible. Re-running Gradle validation is safe.
